Design Exploration

The initial idea I generated during the course discussion is to build a fuel price checker and predictor, I randomly checked some of the datasets and find the fuel price data set with good format and a relatively larger amount. It is an ideal training set to start up a machine learning model. I just complete my machine learning study on Coursera so can’t help to think about how to make use of these data to build a machine learning model.

During the course discussion, the group brainstormed on several topics and some of my classmates found some other related data on Queensland Government data sets, including fuel types, vehicle registration, traffic load, etc. After class, I expanded the topic into “Driving in Queensland”, aimed to provide a one-stop traffic information portal for drivers in Queensland.

Based on the above idea I tried to make a mock-up of the website homepage. The target users can be different types of drivers including car drivers, motorcycle drivers, and truck drivers. However, I later dropped this idea when I find it is hard to justify the user scenario.

Drivers do not have a PC with internet access when driving. The website seems not of use if it’s not designed with a mobile-first experience. As a newbie in web design, I’m not confident to start with a mobile site. So I switched my topic to “weekend planner”.

To start all over again, I checked the Padlet board for the course discussion again. One of the discussion notes talked about some upcoming firework show. This inspired me about thinking about making an event index. Not only limited to fireworks but also other types of events, for example, Art Gallery Events.

To create an easy to use front-end interaction, I’m thinking of some keyword search. From my machine learning course, there’s easy access to achieve this using word embedding. Giving some keywords, the algorithm can return a word representing the most related event name in the event word list. The backend can then search the datasets by using the generated event name.

Thinking of the target user, it can be any ordinary people who don't have a clear clue about what to do this weekend. This website can be helpful for them to get some hints. And thinking about the user scenario, users may want to have a quick answer in less than 10 minutes because it is not something serious and need to spend a lot of time on it.

Since the overall interaction happens within a very short time frame, the UI needs to be as simple as possible. Inspired by Padlet, the sticky notes seems an easy way to add some random keywords.

When designing the mockup, I realize that the website must provide a share function. Often people are spending their weekends with friends, family, or significant other. So the planner needs to offer some flexibility for mobile access when a PC user sharing with someone without a PC for the moment.

After having the website mockup ready, the next step is to organize them into a poster. The left side is introducing the website itself while the right side is telling a story using a WhatsApp chat history.

To not beak the aesthetics, I did not use the ordinary format of a storyboard. Instead, the chat history format is telling the story well enough with sufficient detail for the audience to know how to interact with the website.

site_map and tech_stack

miro.com

After the task allocation, as the lead developer, I defined the architecture of our website with the necessary technologies explained in the following Miro board. Considering the skill set of the team, I decided to use Python, PHP, and MySQL as our backend. HTML, CSS, and JavaScript are used for frontend.

The Miro board is created to explain the idea to my teammates.

After the architecture is defined, next thing is to set up our team zone. As the lead developer, I’m installed python 3.8 with PIP, Pandas, Requests, and MySQL connector libraries for our team zone. Also, I configured MySQL and PHP to make them function in our team zone. Somehow the package provided by UQ zone ‘webproj’ image is problematic for us. Fortunately, the team zone is a Ubuntu system so basically I did the configuration through ‘sudo apt-get’ method.

The backend is developed by me using python. The basic idea is to retrieve the WildNet API data using Python request library, then write them into a CSV file for human reading using the Pandas library, and by the same time write the data into MySQL using MySQL connector library.

I also coded the template for the animal detail page. The key part is to retrieve data from the MySQL database and put them into an HTML table. And I defined the CSS for this page to achieve the aesthetic design.

Stage3 - Final Delivery Action Item

Google Docs

Due to the complexity of the task, I discarded the Gantt diagram. The only tracker now is the spreadsheet. Action items start from 50 are for part C. To buy me more time on the coding part, I assign the planning for testing and documentation to Wei-Ning and Cassie.

After part B, we received some useful feedback. We want to absorb some of them into our MVP. Also, the voting function for the Ranking page is not ready and consider complicated. To complete our final product in time, I hosted a team meeting for scope refinement. The decision is to add an interactive game on our homepage to increase interactivity and to make the vote function a simulator.

The ranking page is to present the voting outcome for different animals and show their ranking. In the part B demonstration, it’s just a static HTML page. However, to make it functional, the ranking of animals needs to be retrieved dynamically from the server. The original design is using MySQL to store all the data including the votes cast. To simplify the solution, I used the d3 library for JavaScript to parse a CSV file on the server. The voting information is stored in the file. The d3 library offers an easy method for me to parse the CSV file and generate HTML elements dynamically.

Also, the HTML elements generated need to have interactions for casting votes and linking to the animal detail pages. The method d3 offered is an async data retrieving method very much like AJAX. Which makes it difficult to simply add interactions like $(“.animal”).click(). The workaround is to add the interactions after the generation of the HTML element itself.

In part B presentation we received feedback saying that the whole website is lacking interaction. The team thinks it would be better to add in more interactive elements. So a “who am I” guessing game is added to the homepage. I coded for this section using pure JavaScript controlling the visibility of different HTML elements.

  • I didn't receive feedback during the contact. But when later I meet up with my group members to discuss our team topic, some of my team members provided a nice point. She says the poster reuses too much of the similar designs, and the placeholder text is not informative enough to show what would be presented on the result page.

But later I decided to add in persona and storyboard. I made an individual page to include the story. However, I did not make any changes to the previous page. This makes the whole poster inconsistent in a sense. I make use of the same mockup as the first page in my user story. So overall the poster gives the viewer a feeling that too much-reused materials appear.

  • The good thing about doing so is that it saves a lot of time. But a strong side-effect is introduced since the over-reused visual elements give the reader a perception that this website is lacking content. It looks like there are no sufficient details to support the concept, or the concept itself has some flaws that I'm trying to avoid.

  • It reminds me that during the website design if I want to add or remove something, better to rethink the integrity of the whole website. Think about the implication that a modification may have on the big picture, before action. This will improve the final work quality. And be cautious about the repeating elements, unless there's a certain level of necessity, do not make something appear twice or more.

  • In conclusion, now I'm thinking of doing better planning and sectioning before doing the work. Using techniques like card sorting and visual organizing to have a more concrete idea before designing. List out all the contents I want to put on the website, and try to categorize in a logical order, reduce the recurrence. And of course, before making any modification, think about the big picture again, to keep the consistency of the whole design.

  • The action plan for the future if I encounter a similar situation would be when adding on more elements to the existing design, find out the contents before doing visual organization. Do a mock-up and examine with the existing part, to keep integrity, and then make the design according to the mock-up.
